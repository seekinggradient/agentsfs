# agentsfs adopting (bring an existing vault in)

Copy this to your agent when someone wants to adopt an existing folder of notes — an Obsidian vault, a Notes directory, a pile of markdown — replacing `<PATH>`:

> You are adopting the folder at `<PATH>` into an agentsfs. The goal is to make it healthy and self-describing **without rewriting anything the user already wrote**. Read `<PATH>/AGENTS.md` first (run `afs init <PATH>` first if there is none) — it is the contract. Then:
>
> 1. **Orient.** Run `afs tree <PATH>` and `afs doctor <PATH>` (add `--json` for structured output) to see the shape and the current findings. Skim the top-level directories and a few notes so you understand what this vault *is* before you touch anything.
> 2. **Interview briefly.** Ask the user — in a sentence or two, not a survey — which areas are *active knowledge* (things they and their agents will read, extend, and rely on), which are *personal chronology* (a diary, daily notes, a journal they keep by hand), and which are *media or archive* (attachments, images, exports, old material kept for the record). You are classifying their existing folders, not proposing new ones.
> 3. **Mark the reserved roles.** The session journal and scratch space are declared by a marker, not a name (`agentsfs_role: journal` / `agentsfs_role: scratch` in a directory's `INDEX.md`). **Never claim an existing personal directory as the journal** — its consume-and-delete semantics would destroy the user's diary. If they want a session journal, create or designate one *with them* (the default `agent-journal/` is safe). Same for scratch.
> 4. **Declare collections.** For each *personal-chronology*, *media*, and *archive* directory, add `agentsfs_role: collection` to its `INDEX.md` frontmatter — or, if it has none, create a minimal `INDEX.md` with a truthful one-line `description:` plus the marker. A collection is described *collectively* by that INDEX: doctor stops asking for per-file descriptions, per-subdirectory INDEX files, and link health beneath it, and it is never emptied or deleted. This is how a diary or an attachments folder joins the instance without being pulled apart. Keep descriptions of personal material short and neutral.
> 5. **Then genuinely annotate the active areas** — this is the real work, and it is additive only. For the directories the user called active knowledge: add a one-line `description:` to each markdown file (what the file is *for*, in a neutral tone), create the missing `INDEX.md` files (with their own `description:`, plus a line for each file that can't describe itself), and repair the dead links doctor flags there. Use `[[wikilinks]]` where a real entity is named.
> 6. **Additive-only, always.** Never modify the body of an existing note. Never rename, move, merge, or split during adoption — that comes later, once the user trusts the process. You are adding frontmatter and INDEX files around what exists, nothing more.
> 7. **Small, reviewable commits.** Commit per area or per pass — "mark Diary/ and Attachments/ as collections", "describe the Projects/ notes" — with messages that say what you annotated. `git add -A . && git commit` from `<PATH>`.
> 8. **`afs doctor` is the progress meter.** Re-run it after each pass; the findings shrink as you go. Target is healthy — and staying healthy as the user keeps adding notes. When it is clean, hand back a short summary of what you marked, what you described, and what (if anything) you left for the user to decide.
>
> Hard rules: additive only — no note body is edited, nothing is renamed, moved, merged, or deleted during adoption. Respect privacy — descriptions of personal material stay short and neutral, and a personal directory becomes a collection, never the journal. When you don't know what a file is *for*, ask or write "unknown"; do not guess.
